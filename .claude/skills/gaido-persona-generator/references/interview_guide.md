# 例文ベース壁打ちガイドライン

## 概要

ペルソナ生成のStep 3で使用するガイドライン。
エゴグラム特性リストだけでは「トーン制御」しかできないため、**思考スタイル・問い返し軸・振る舞いルール**をAIとの壁打ちで引き出す。

## 設計原則

### なぜ「例文ベース」なのか

- 抽象的な質問（「何を重視しますか？」）より、具体的な例文（「この人ならこう言いそうですか？」）の方が正確な情報が出る
- 自己申告と実際の人格にはズレがある。例文を見て「ここが違う」と言ってもらうことでズレを補正する
- エゴグラムだけの例文を出して「違う」と言われたら、**その差分が暗黙知**

### 進め方の原則

- **質問数は固定しない**。納得感が出るまで繰り返す
- **例文への反応が次の質問を決める**。固定の質問リストではなく適応的に進める
- **「違う」と言いやすい形式にする**。○×で答えられるよう具体的に提示する
- **各段階で中間確認を挟む**。全部聞いてから一発生成ではなく、途中で軌道修正する
- **例文への反応がすべて「それに近い」で済んだ場合でも、別のシーンで追加確認する**。1つのシーンでの一致は、他のシーンでも一致するとは限らない。早期に打ち切らず、複数のシーンで検証すること

## 壁打ちの3段階

### 段階1: 判断基準の引き出し

**目的**: persona_template.mdの「思考スタイル」セクションに相当する情報を得る。

#### 実在の人物の場合

エピソードから判断基準を抽出する。

**開始質問（自由記述）**:
- 「その人が過去に下した判断で、最も印象的なものを教えてください。何をYesにし、何をNoにしましたか？」
- 「その人が怒る（不満を示す）のはどんなときですか？ 逆に、素直に賛同するのはどんなときですか？」

**例文提示（シーンベースで動的生成）**:

Step 0で取得した**用途 + 役職・立場**からその人が日常的に遭遇する具体的なシーンを生成し、そのシーンでの反応をエゴグラム特性に基づいて例文として提示する。抽象的な性格描写より、具体的なシーンの中での発言のほうがユーザーが「合っている/違う」を判断しやすい。

シーン生成ルール:
1. **用途**が文脈を絞る（営業案件→案件レビュー系、提案書→レビュー系、顧客理解→商談系）
2. **役職・立場**がシーンを具体化する（部長→部下からの報告、CTO→技術選定の相談、等）
3. 1つのエゴグラム特性につき**1シーン1例文**を提示する

シーン生成の例（用途 + 役職の組み合わせ）:

| 用途 | 役職 | シーンの例 |
|------|------|-----------|
| 営業案件の壁打ち | 営業部長 | 「部下が新規案件の受注可否を相談に来た場面」 |
| 営業案件の壁打ち | 本部長 | 「四半期レビューで案件ポートフォリオを確認している場面」 |
| 提案書の壁打ち | CTO | 「提案書の技術構成パートをレビューしている場面」 |
| 提案書の壁打ち | 経理部長 | 「提案書のコスト見積もりセクションを確認している場面」 |
| 汎用的な壁打ち | PM | 「プロジェクトの進捗会議で問題報告を聞いている場面」 |
| 汎用的な壁打ち | 営業部長 | 「部下が方針について相談に来た場面」 |
| 顧客理解 | 顧客の情シス部長 | 「ベンダーからシステム提案を受けている場面」 |

例文生成ルール（シーン + エゴグラム特性）:
- A高 → シーン内で数字・根拠を確認する発言。例（営業部長 + 案件相談）: 『で、利益率は何パーセント？工数の根拠は誰が出したの？』
- CP高 → シーン内で曖昧さを許さない発言。例（本部長 + 四半期レビュー）: 『この案件、前回も同じこと言ってたよね。何が変わったの？』
- NP高 → シーン内で相手を配慮しつつ聞く発言。例（PM + 進捗会議）: 『大変だったね。ちなみに、リカバリプランはもう考えてる？』
- FC高 → シーン内で前のめりになる発言。例（CTO + 技術レビュー）: 『この技術スタック面白いね！PoC作ってみようよ！』
- AC低 → シーン内で忖度せず指摘する発言。例（経理部長 + コスト確認）: 『正直、この見積もりは甘いと思う。根拠を見せて』

**ユーザーの反応に基づく適応**:
- 「それに近い」→ その方向で次の例文を出し、より具体的に詰める
- 「ちょっと違う」→ どこが違うか聞き、エゴグラムとの差分を記録する
- 「全然違う」→ 別の角度から例文を生成し直す

**中間確認**: 2-3個の例文確認が終わったら、「ここまでの話をまとめると、この人の思考スタイルはこんな感じですか？」と箇条書きで提示する。

#### 架空の人物の場合

AIがロールの典型像をベースに提案し、ユーザーに調整してもらう。

**開始質問**:
- 「{役職名}として最も重視する価値観は何ですか？」（選択式: 収益性 / 技術的品質 / スピード / チームの成長 / 顧客満足）
- 「どんな提案・報告を却下しますか？」（選択式: 根拠が弱い / コストが見合わない / リスクが高い / 戦略に合わない）

**例文提示**: 用途＋役割から具体的なシーンを生成し、エゴグラム＋ユーザーの回答からそのシーンでの反応を例文として提示する。架空の人物は実際のエピソードがないため、シーンがないと「合っている/違う」を判断する手がかりがない。シーンを提示したうえで「もっとこうしたい」を聞く。

### 段階2: 問いの軸の定義

**目的**: persona_template.mdの「問い返し軸」セクションに相当する情報を得る。

#### 用途別の質問アプローチ

**営業案件の壁打ち**:
- honbucho.mdの5軸（数字/将来性/勝算/リスク/戦略）をベースに提案
- 「この人がレビューするとき、こういう順番で見そうですか？」と例文で確認
- 例: 「まず利益率を確認 → 次に競合との差別化 → 最後にリスク。こんな感じですか？」

**提案書の壁打ち**:
- 提案書の評価軸を提案: ストーリーの一貫性 / データの裏付け / 顧客課題への共感 / 差別化ポイント / 実現性
- 「この人が提案書を見たとき、最初に確認するのはどこですか？」

**汎用的な壁打ち相手**:
- 「この人の専門領域は何ですか？」
- 「議論の中で必ず確認する観点は？」
- 「OK/NGの判断基準を教えてください」

**顧客理解（営業支援）**:
- 「この顧客が意思決定時に最も重視することは？」
- 「この顧客に刺さるキーワードや避けるべきワードはありますか？」

#### 例文提示

段階1で得た思考スタイルをもとに、**具体的なシーンの中で**「この人ならこういう観点で見そうですか？」と提示する。

例（営業部長 + 案件レビュー）:
> シーン: 「部下が新規SIer案件の提案書を持ってきて説明している場面」
> 「ここまでの話から、この人は以下の順番で確認しそうですか？」
> ```
> 1. まず『利益率は？　工数の根拠は？』と数字を確認する
> 2. 次に『競合は？　うちが勝てる理由は？』と勝算を確認する
> 3. 最後に『この案件、次につながるの？』と将来性を確認する
> ```
> 「リスクはそんなに見ない。代わりに〜を見る、という場合は教えてください」

**中間確認**: 軸が固まったら「この人の問い返し軸はこれで合ってますか？」と一覧で提示する。

### 段階3: 振る舞いルールの確認

**目的**: persona_template.mdの「問い返しのルール」セクションに相当する情報を得る。

#### 確認ポイント

以下の各ポイントについて、**用途＋役職から生成した具体的なシーンの中で**例文を提示して確認する。シーンと例文はStep 0の情報に応じて動的に生成すること。

各ポイントで確認すべき**観点**:

**議論の進め方**:
- シーンを提示し、「この人はこの場面でどう議論を進めますか？」と聞く
- 観点: 1つの論点を掘り下げてから次へ / 全体を俯瞰してから深掘り

**不十分な回答への反応**:
- シーンの中で曖昧な回答が返ってきた状況を提示する
- 観点: 即座に突っ込む / 柔らかく掘り下げる / 角度を変えて探る

**OK判定の基準**:
- シーンの中で説明が終わった状況を提示する
- 観点: 数字・事実が具体的に示されたとき / 方向性が合っていれば細部は問わない / 対処方針が現実的で実行可能なとき

**OKのときの反応**:
- シーンの中で納得した状況を提示する
- 観点: 端的に承認して次の指示 / 感情的に反応 / 淡々と承認

**フィードバックスタイル**:
- シーンの最後にフィードバックする状況を提示する
- 観点: 問題点のみ指摘 / 良い点と改善点の両方 / アクションリスト形式

**中間確認**: 「この人の対話スタイルをまとめると、こんな感じですか？」と要約して提示する。

## エゴグラムからの例文生成ルール

### シーン生成の原則

例文は必ず**具体的なシーンの中での発言**として提示する。抽象的な性格描写（「論理的に問う」）ではなく、特定の場面での具体的な発言（「四半期レビューで『この案件の利益率は？根拠は？』と問う」）として表現する。

シーン生成の優先順位:
1. **用途 + 役職**から生成（最も具体的）
2. **役職のみ**から生成（用途が汎用的な場合のフォールバック）
3. **用途のみ**から生成（役職が曖昧な場合のフォールバック。例: 「厳格なレビュアー」→ レビューの場面）

### 組み合わせパターン

主要な組み合わせに対して、シーン内での例文のトーンを調整する:

| パターン | シーン内での例文のトーン |
|---------|------------|
| A高 + CP高 | 「なぜ？」「根拠は？」と鋭く論理的に問う |
| A高 + NP高 | 「理由を教えてもらえる？」と丁寧だが論理的に問う |
| CP高 + AC低 | 忖度なく基準に照らして即断する |
| NP高 + FC高 | 共感的で乗り気になりやすいが、心配もする |
| A高 + FC低 | 冷静で淡々と事実確認する |
| CP低 + NP高 | 厳しい指摘はせず、フォロー・サポート寄り |

### NOT-OKの扱い

例文にはOK特性だけでなく、NOT-OK特性も反映させる。ただし**NOT-OKを直接的に表現するのではなく、行動として出る形**で反映する。

- NOT-OK「物事にシビアすぎる」→ 例文では「些細な数字のズレも見逃さない」として反映
- NOT-OK「出しゃばりすぎる」→ 例文では「聞かれていないことまでアドバイスする」として反映
