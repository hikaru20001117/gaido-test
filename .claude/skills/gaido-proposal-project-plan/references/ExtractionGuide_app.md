# 固有情報抽出ガイド（アプリケーション開発）

提案書・提供資料からプロジェクト計画書に必要な固有情報を抽出する際のガイド。
Webシステム・業務アプリケーション・スマートフォンアプリ等のソフトウェア開発案件に適用する。

## 抽出カテゴリ一覧

| # | カテゴリ | 抽出内容 |
|---|---------|---------|
| 1 | プロジェクト基本情報 | プロジェクト名・顧客名・発注背景・目的・ゴール・期待効果・PM承認者（スポンサー）・PM権限範囲 |
| 2 | スコープ | 対象機能・画面・API・連携システム・DB対象・非機能要件・対象外範囲・制約事項 |
| 3 | スケジュール | フェーズ定義（要件定義〜リリース）・期間・開始/終了日・主要マイルストーン・制約日程 |
| 4 | 体制 | 顧客側窓口・承認者・業務担当者・情報システム部門・ベンダー側PM・テクニカルリード・担当者 |
| 5 | 成果物 | 設計書・ソースコード・テスト結果・リリース手順書・移行手順書等の一覧・形式・検収条件・担当 |
| 6 | リスク | 想定リスク・影響度・発生確率・対策方針（要件変更・セキュリティ・データ移行・リソース等） |
| 7 | 予算・費用 | 契約総額・費用区分（開発費・ライセンス費・インフラ費等）・支払い条件（記載がある場合のみ） |
| 8 | 引渡し基準・品質管理方針 | 工程完了条件・テスト合格基準・レビュー計画・引渡し判定基準 |
| 9 | コミュニケーション管理 | 会議体一覧（種類・頻度・主催・参加者）・使用ツール・ファイル共有方法・報告ルール |
| 10 | 変更管理方針 | 変更管理対象（機能追加・仕様変更・スケジュール変更）・変更フロー・文書化ルール |
| 11 | 顧客側作業・協力事項 | テスト参加・データ提供・環境準備・レビュー対応・本番承認等、顧客が担うタスク一覧 |

## 抽出のポイント（アプリ開発特有）

### カテゴリ 2（スコープ）で確認すること

- **機能一覧**: 提案書の機能要件・画面一覧・ユースケース図から抽出
- **連携システム**: 既存システム・外部サービス（決済・認証・メール配信等）との連携有無
- **DB対象**: 新規DB設計か既存DB流用かを明記
- **対象外**: インフラ設定・ハードウェア調達・ユーザー研修が含まれるか確認

### カテゴリ 3（スケジュール）で確認すること

- アプリ開発の標準フェーズ（要件定義→基本設計→詳細設計→製造→テスト→リリース）が提案書に記載されているか
- フェーズ名が異なる場合（例: 「開発フェーズ」まとめ）は実態を確認して分解する
- データ移行・並行運用期間・カットオーバー日が明記されているか

### カテゴリ 5（成果物）で確認すること

アプリ開発で標準的に生成される成果物:
- 要件定義書
- 基本設計書（画面設計書・DB設計書・API仕様書）
- 詳細設計書
- ソースコード・設定ファイル
- 単体テスト仕様書・結果
- 結合テスト仕様書・結果
- 総合テスト仕様書・結果
- リリース手順書
- データ移行手順書・移行結果（移行がある場合）
- 運用マニュアル（含まれる場合）

提案書に記載がない成果物は「TBD」として「不足情報」に追記する。

### カテゴリ 6（リスク）で確認すること

提案書に記載がなくても、以下は必ずリスクとして追記を検討する:
- 要件の変更・追加（発生頻度が高い）
- 外部システム連携の仕様確定遅れ
- テスト工数の見積もり不足
- セキュリティ要件の後出し

### カテゴリ 8（品質管理）で確認すること

- テスト合格基準（合格率100%・バグ件数基準等）が記載されているか
- コードレビュー・静的解析の実施有無
- セキュリティ診断・脆弱性テストの要否

## 抽出ルール

- 各カテゴリについて、提案書・資料の**原文に忠実**に抽出する
- 具体的な数値・固有名詞・日付・社名を必ず含める（抽象的な表現のみは禁止）
- 資料に記載がないカテゴリは「記載なし」と明記してスキップしない（カテゴリ自体は省略しない）
- 複数の資料がある場合は矛盾がないか確認し、最新・詳細な記載を優先する
