### スライド 3: プロジェクト概要（憲章）

- **種別**: mixed
- **メッセージ**: {このプロジェクトで解決するシステム課題・ゴール・PM権限を明示する}
- **参照元**: プロジェクト基本情報
- **内容**:
  - 発注背景・現状課題: {例: 既存システムの老朽化による業務非効率、法改正対応の必要性}
  - プロジェクト目的: {例: 新業務システムの構築による業務効率化}
  - ゴール・達成基準: {例: 〇〇機能を持つWebシステムを{リリース日}にリリースする}
  - 期待効果: {例: 処理工数〇%削減、エラー率〇%低下}
  - 対象システム・サービス: {システム名}
  - PM承認者（スポンサー）: {スポンサー氏名・役職}
  - PM権限範囲: {予算決裁上限・スコープ変更承認権限}
- **図表**: コンテキスト図（draw.io DrawioBuilder）
  - 左エリア（`STYLE_GROUP_DASHED_RED`）: 現行システム名 + 課題ラベル（`STYLE_LABEL_SMALL`）
  - 右エリア（`STYLE_GROUP_GREEN`）: 新システム名 + 期待効果ラベル（`STYLE_LABEL_SMALL`）
  - 中央エッジ（`add_edge`、ラベル: 「プロジェクト実施」）: 左→右を接続
  - 保存先: `assets/context_diagram.drawio` / `assets/context_diagram.png`

### スライド 4: スコープ

- **種別**: mixed
- **メッセージ**: {本プロジェクトで開発・対応する範囲を機能・作業・対象外の観点で明確にする}
- **参照元**: スコープ（カテゴリ2）
- **内容**:
  - **プロダクト・スコープ（何を作るか）**:
    - {機能/画面名1}: {概要}
    - {機能/画面名2}: {概要}
    - 連携システム: {連携先・内容}
  - **プロジェクト・スコープ（何をするか）**:
    - 要件定義・基本設計・詳細設計・製造・テスト・リリース・移行
    - 顧客レビュー対応・議事録作成・定例会運営
    - 納品ドキュメント整備
  - **対象外範囲・制約事項**:
    - {対象外1（例: インフラ設定は別途対応）}
    - {制約事項1}
  - **引渡し基準（検収条件）**:
    - 納品物のレビューが完了しており、すべて納品済みであること
    - 試験項目がすべて消化済みであること（未消化・NGなし）
    - プロジェクト内で発生した課題がすべて解決していること
    - 顧客の承認が取得済みであること
    - {プロジェクト固有の検収条件があれば追記}
- **図表**: スコープ境界図（draw.io DrawioBuilder）
  - 対象スコープ枠（`STYLE_GROUP_DASHED_RED`）: `STYLE_SCOPE_MARKER` で「★ 構築対象スコープ」表示 + 主要機能・API・DB対象をノードで配置
  - 対象外エリア（`STYLE_GROUP_GRAY`）: インフラ・研修等、対象外の作業をノードで配置
  - 外部連携先（`STYLE_EXTERNAL`）: 連携する外部システムをスコープ枠外に配置し `add_edge` で接続
  - 保存先: `assets/scope_diagram.drawio` / `assets/scope_diagram.png`

### スライド 12: 変更管理方針

- **種別**: mixed
- **メッセージ**: {変更管理対象と変更要求発生時のフローを明示する}
- **参照元**: 変更管理方針（カテゴリ10）
- **内容**:
  - **変更管理対象**:

| 対象 | 変更管理対象 | 主な成果物 |
|------|------------|-----------|
| スコープ | 機能追加・削除・仕様変更 | プロジェクト計画書・要件定義書 |
| スケジュール | マイルストーンの変更 | プロジェクト計画書・WBS |
| その他 | 体制変更・業務要件の追加 | 体制図・要件定義書 |

  - **変更管理フロー**: 変更要求発生 → PM記録・判定 → 影響範囲調査（工数・スケジュール・費用） → 変更内容・見積確認 → プロジェクト管理責任者承認 → 変更反映
  - **文書化ルール**: 変更要求に関わる内容は必ずやり取りを文書化して顧客に共有する
- **図表**: 変更管理フロー図（draw.io DrawioBuilder）
  - 6ノードを横一列に配置: 変更要求発生 / PM記録・判定 / 影響範囲調査 / 顧客確認 / 承認 / 変更反映
  - 各ノードを `add_edge`（`orthogonalEdgeStyle`）で左→右に接続
  - 保存先: `assets/change_flow.drawio` / `assets/change_flow.png`
