# アプリケーション開発向けプロジェクト計画書テンプレート

プロジェクト計画書作成手引き（情通プロジェクト統括部 2024/3版）に基づく構成。
Webシステム・業務アプリケーション・スマートフォンアプリ等のソフトウェア開発案件に適用する。

## 各スライドの内容ガイド

### スライド 1: 表紙

- プロジェクト名（正式名称）
- 顧客名（正式社名）
- ベンダー名
- 作成日・バージョン番号
- ドキュメント管理番号（任意）

### スライド 3: プロジェクト概要（憲章）

アプリ開発案件では業務課題・システム老朽化・新機能ニーズ等を中心に記載する。

- 発注背景・現状課題（例: 既存システムの老朽化、業務効率化ニーズ、法改正対応）
- プロジェクト目的・ゴール（例: 新システムリリース、業務工数〇%削減）
- 期待効果・KPI（例: 処理時間短縮、ユーザー操作性向上、エラー率低下）
- 対象システム・サービス名
- PM承認者（スポンサー）: {承認者氏名・役職}
- PM権限範囲: {予算決裁上限・スコープ変更承認権限}

**図表: コンテキスト図（draw.io DrawioBuilder）**

現状（課題）と新システム（解決後）を対比する2ブロック構成。

| エリア | スタイル | 内容 |
|-------|---------|------|
| 左: 現状 | `STYLE_GROUP_DASHED_RED` | 現行システム名ノード + 課題ラベル |
| 右: 新システム | `STYLE_GROUP_GREEN` | 新システム名ノード + 期待効果ラベル |
| 中央矢印 | `add_edge`（ラベル: 「プロジェクト実施」） | 左ブロックから右ブロックへ |

保存先: `assets/context_diagram.drawio` / `assets/context_diagram.png`

### スライド 4: スコープ

**プロダクト・スコープ（何を作るか）**
- 対象機能・画面一覧（主要ユーザーストーリー単位）
- API・連携システム（内部連携・外部サービス連携）
- DB・データ設計対象
- 非機能要件（性能・セキュリティ・可用性）

**プロジェクト・スコープ（何をするか）**
- 要件定義・基本設計・詳細設計・製造・テスト・リリース・移行
- 顧客レビュー対応・議事録作成・定例会運営
- ドキュメント整備・納品作業

**対象外範囲・制約事項**
- インフラ調達・サーバー設定（別途インフラ案件として対応する場合）
- ハードウェア購入・ネットワーク工事
- 顧客社内教育・操作研修（別途対応の場合）
- 対象機能以外の既存システム機能改修

**図表: スコープ境界図（draw.io DrawioBuilder）**

対象スコープと対象外を視覚的に区別する境界図。

| エリア | スタイル | 内容 |
|-------|---------|------|
| 対象スコープ枠 | `STYLE_GROUP_DASHED_RED` | 「★ 構築対象スコープ」ラベル（`STYLE_SCOPE_MARKER`）+ 機能・連携システムをノードで配置 |
| 対象外エリア | `STYLE_GROUP_GRAY` | 対象外の機能・作業をノードで配置 |
| 外部連携先 | `STYLE_EXTERNAL` | 連携する外部システムを対象スコープ外に配置し、エッジで接続 |

保存先: `assets/scope_diagram.drawio` / `assets/scope_diagram.png`

**引渡し基準（検収条件）**
- 納品物のレビューが完了しており、すべて納品済みであること
- 試験項目がすべて消化済みであること（未消化・NGなし）
- プロジェクト内で発生した課題がすべて解決していること
- 顧客の承認が取得済みであること
- {プロジェクト固有の検収条件があれば追記}

### スライド 11: リスク管理方針

アプリ開発案件で発生頻度の高いリスクを中心に記載する。提案書に記載のリスクは上書きまたは追記すること。

| # | リスク内容 | 影響度 | 発生確率 | 対策方針 |
|---|-----------|--------|---------|---------|
| 1 | 要件の変更・追加 | 高 | 高 | 変更管理フローを徹底・スコープ合意を文書化 |
| 2 | テスト工数の見積もり不足 | 高 | 中 | バッファを確保・早期にテスト計画を確定 |
| 3 | 外部システム連携の仕様不明確 | 高 | 中 | 連携先との仕様確認を要件定義フェーズで完了 |
| 4 | セキュリティ脆弱性の発覚 | 高 | 低 | セキュリティレビューを設計・テスト工程に組み込む |
| 5 | データ移行の整合性問題 | 中 | 中 | 移行リハーサルを本番前に複数回実施 |
| 6 | 主要メンバーの離脱 | 高 | 低 | ナレッジをドキュメント化・複数担当者で知識共有 |
| {追加リスク} | | | | |

**リスク管理の運用方針**
- 週次定例でのリスク定期レビュー
- エスカレーション基準: 影響度「高」かつ発生確率「高」の場合、翌営業日以内にPM→プロジェクト管理責任者へ報告

### スライド 12: 変更管理方針

**変更管理対象**

| 対象 | 変更管理対象 | 主な成果物 |
|------|------------|-----------|
| スコープ | 機能追加・削除・仕様変更 | プロジェクト計画書・要件定義書 |
| スケジュール | マイルストーンの変更 | プロジェクト計画書・WBS |
| その他 | 体制変更・業務要件の追加 | 体制図・要件定義書 |

**変更管理フロー**
変更要求発生 → PM記録・判定 → 影響範囲調査（工数・スケジュール・費用） → 顧客確認 → プロジェクト管理責任者承認 → 変更反映

**図表: 変更管理フロー図（draw.io DrawioBuilder）**

上記フローを矩形ノード＋エッジで横並びに表現する。

| 要素 | 内容 |
|-----|------|
| ノード（6個） | 変更要求発生 / PM記録・判定 / 影響範囲調査 / 顧客確認 / 承認 / 変更反映 |
| エッジ | 各ノードを `orthogonalEdgeStyle` で左→右に接続 |
| スタイル | デフォルト矩形（`add_node` のデフォルト） |

保存先: `assets/change_flow.drawio` / `assets/change_flow.png`
